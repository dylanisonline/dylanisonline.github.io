Organizational Unit Design


Before creating users, security groups, or administrative permissions, the first step was designing an Organizational Unit structure for the kojima.test domain. Getting this right upfront matters — the OU structure determines how Group Policy is applied, how delegation is scoped, and how cleanly the environment scales as new users, machines, and roles are added.


The goal was to create a structure that would support:

  • Clear separation between standard and privileged accounts
  • Role-based access control via the AGDLP model
  • Delegated Active Directory administration without requiring Domain Admin rights for routine tasks
  • Targeted Group Policy application scoped to the right objects
  • Department-specific workstation management
  • Future expansion without structural rework


Final Design

The completed structure separates identities, computers, roles, and resource-access groups into five distinct top-level Organizational Units.



Top Level OUs:

OU Purpose
Kojima Users Contains standard employee accounts
Kojima Admins Contains dedicated privileged accounts
Kojima Computers Contains domain-joined workstations and servers
Kojima Roles Contains Global Security Groups representing job roles and administrative task permissions
Kojima Resources Contains Domain Local Groups representing delegated permissions and resource access


Identity OUs

The identity OUs separate standard employee accounts from privileged administrative accounts.


Kojima
├── Kojima Users
│   ├── IT                  (Standard IT user accounts)
│   ├── Development         (Standard development user accounts)
│   └── Operations          (Standard operations user accounts)
│
└── Kojima Admins
    ├── Domain Admins       (Domain-level administrative accounts)
    ├── Server Admins       (Server administrative accounts)
    └── Workstation Admins  (Workstation administrative accounts)


Standard and privileged accounts are stored separately so they can be managed independently. Administrative accounts receive stricter Group Policies, logon restrictions, auditing requirements, and privileged-access controls without those settings affecting standard user accounts.

Each privileged account represents a specific administrative scope. A single person may hold multiple privileged accounts — one per scope — rather than a single elevated account used for all tasks. This enforces least privilege at the identity level.


Computer OUs

The computer OUs separate member servers from end-user workstations.


Kojima Computers
├── Servers                 (Domain-joined member servers)
├── Staging                 (Newly joined machines pending OU assignment)
└── Workstations            (Domain-joined user endpoints)
    ├── IT                  (IT-assigned workstations)
    ├── Development         (Developer workstations)
    └── Operations          (Operations workstations)


Servers and workstations are stored separately because they require different security baselines, administrative controls, and Group Policy settings.

The Staging OU holds newly domain-joined machines before they are moved to their target OU. This prevents unvetted machines from immediately receiving production GPOs and gives administrators a window to configure and validate each machine before it enters the managed environment.

The workstation structure is divided by department so that policies can be applied only where they are needed. For example:

  • IT workstations may receive administrative utilities and support tooling.
  • Development workstations may receive approved local administrator access and development tooling.
  • Operations workstations may receive a more restrictive standard-user configuration.


Roles OUs

The Roles OUs contain Global Security Groups. These groups never hold permissions directly — they control membership. Permissions flow through the Resource OUs via the AGDLP chain described in the Task Groups section.

There are three distinct sub-OUs under Kojima Roles, each serving a different purpose:


Kojima Roles
├── Admin Roles             (Scoped privileged identity groups)
│   ├── Kojima-GG-IT-Role-SysAdmin-WS
│   ├── Kojima-GG-IT-Role-SysAdmin-Server
│   └── Kojima-GG-IT-Role-SysAdmin-Domain
│
├── Task Roles              (Administrative task permission groups)
│   ├── Kojima-GG-User-Management
│   ├── Kojima-GG-Computer-Management
│   ├── Kojima-GG-GPO-Management
│   ├── Kojima-GG-Group-Management
│   └── Kojima-GG-OU-Management
│
└── Department Roles        (Standard organizational membership groups)
    ├── IT
    │   └── Kojima-GG-IT-Staff
    ├── Development
    │   └── Kojima-GG-Development-Staff
    └── Operations
        └── Kojima-GG-Operations-Staff


Admin Roles represent elevated administrative identities scoped by account tier. A Sr. Systems Engineer holds three separate Role GGs — one for workstation-level access, one for server-level access, and one for domain-level access. This ensures each privileged account carries only the permissions its scope requires.

Task Roles represent specific administrative functions within Active Directory — managing users, computers, GPOs, groups, and OUs. No user or admin account is a direct member of a Task GG. Instead, Admin Role GGs are nested into the appropriate Task GGs, and Task GGs are nested into Domain Local permission groups. This compositional model means adding a new role requires only updating group membership, not redefining permissions.

Department Roles represent standard organizational membership. A standard user account belongs to exactly one Department Role GG corresponding to their team. These groups are used for email distribution, resource access scoping, and future application role assignments.


Resource OUs

The Resource OUs contain Domain Local Security Groups. These groups hold the actual delegated ACLs and share/NTFS permissions. Nothing is ever assigned directly to a user or a Global Group — access always flows through a Domain Local group.


Kojima Resources
├── Domain Access           (Delegated Active Directory permissions)
│   ├── Kojima-DL-Users-Manage
│   ├── Kojima-DL-Computers-Manage
│   ├── Kojima-DL-GPO-Manage
│   ├── Kojima-DL-Groups-Manage
│   └── Kojima-DL-OUs-Manage
│
├── Computer Access         (Workstation and server management access)
│   └── Kojima-DL-Workstations-LocalAdmin
│
└── File Shares             (NTFS and share-permission groups)
    └── Kojima-DL-FS01-IT-Modify


Domain Local groups are scoped to a single permission target — one OU, one share, one resource. This keeps the blast radius of any delegation small and auditable. When access needs to be revoked, removing a Global Group from a Domain Local group is sufficient — no ACL editing required.
