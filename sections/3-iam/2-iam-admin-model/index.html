Tiered Admin Model


In many environments, administrators operate with a single elevated account used for everything — checking email, browsing the web, resetting passwords, and making domain-wide changes. This is a significant security risk. If that account is compromised, an attacker immediately has access to everything that account can touch.

The Kojima lab addresses this with a tiered admin model. Each person is assigned multiple accounts, each scoped to a specific level of access. Elevated accounts are only used when the task requires that level of elevation — never for routine work.


The Problem With a Single Admin Account

A single admin account creates several compounding risks:

  • Blast radius — a compromised credential exposes every system that account can reach
  • No audit clarity — logs show the same account performing both routine and privileged actions, making forensic analysis harder
  • GPO targeting — applying stricter security policies to privileged accounts is difficult when standard and admin work happen from the same identity
  • Least privilege violations — domain-wide permissions are active even when the task only requires local workstation access

A tiered model eliminates these problems by ensuring no single account carries more access than its specific purpose requires.


The Four Account Tiers

Each person in the Kojima domain is provisioned with up to four accounts depending on their role. Each account lives in a dedicated OU and carries only the permissions its scope demands.


Tier Account Format OU Purpose
Standard firstname.lastname Kojima Users Day-to-day work — email, files, applications. No elevated permissions.
Workstation Admin lastname-ws Kojima Admins/Workstation Admins Local administrator rights on domain-joined workstations only.
Server Admin lastname-sa Kojima Admins/Server Admins Administrative access to member servers. No domain-level AD permissions.
Domain Admin lastname-da Kojima Admins/Domain Admins Domain-wide Active Directory tasks — managing users, groups, GPOs, and OUs.


The standard account is used for all routine work. Elevated accounts are used only when the task explicitly requires that scope. A domain admin account should never be used to browse the web. A workstation admin account should never be used to modify AD objects.


How Tiers Connect to Permissions — AGDLP

Permissions in the Kojima domain are never assigned directly to a user account. Instead they flow through a four-layer chain known as AGDLP:

  • A — Account (the user or admin account)
  • G — Global Group (the Role GG representing who the account is)
  • DL — Domain Local Group (the Permission group holding the actual ACL)
  • P — Permission (the delegated right or resource access)

The Task GG layer sits between G and DL as a compositional middle tier — it groups related permissions so that multiple Role GGs can share the same task permissions without duplicating delegations.

The full chain looks like this:


Domain Admin account example:

lastname-da
    └── Kojima-GG-IT-Role-SysAdmin-Domain     (Role GG — who this account is)
            └── Kojima-GG-User-Management      (Task GG — what task is permitted)
            │       └── Kojima-DL-Users-Manage (DL — holds the delegated ACL)
            │               └── [Delegated permission on Kojima Users OU]
            │
            └── Kojima-GG-Computer-Management
            │       └── Kojima-DL-Computers-Manage
            │               └── [Delegated permission on Kojima Computers OU]
            │
            └── Kojima-GG-GPO-Management
            │       └── Kojima-DL-GPO-Manage
            │               └── [Delegated permission in GPMC]
            │
            └── Kojima-GG-Group-Management
            │       └── Kojima-DL-Groups-Manage
            │               └── [Delegated permission on group objects]
            │
            └── Kojima-GG-OU-Management
                    └── Kojima-DL-OUs-Manage
                            └── [Delegated permission on OU objects]


Workstation Admin account example:

lastname-ws
    └── Kojima-GG-IT-Role-SysAdmin-WS         (Role GG — scoped to workstations only)
            └── Kojima-GG-Computer-Management  (Task GG — computer management only)
                    └── Kojima-DL-Computers-Manage
                            └── [Delegated permission on Kojima Workstations OU]


The workstation admin account nests into a single Task GG. The domain admin account nests into all five. The account scope determines which Role GG it belongs to, and the Role GG determines which Task GGs it feeds into. Permissions are defined once and composed — never duplicated.


Why This Model Scales

When a new role is added — such as a HelpDesk analyst — only two things need to happen:

  1. Create a Role GG for that title (Kojima-GG-IT-Role-HelpDesk)
  2. Nest it into the appropriate Task GGs (e.g. Kojima-GG-User-Management only)

No new delegations. No new DL groups. No ACL editing. The permission infrastructure already exists — the new role simply plugs into it.

This is the core advantage of the AGDLP model over direct permission assignment. The more roles and resources the environment grows to include, the more this separation pays off.
